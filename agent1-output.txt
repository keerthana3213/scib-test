## UI COMPONENT ARCHITECTURE

1. JIRA REQUIREMENT SUMMARY

Story Description:
AS a parent or guardian I WANT to allocate and manage funds for my child's youth account SO THAT I can teach financial responsibility while maintaining visibility and control over spending

Acceptance Criteria:
- Scenario 1 – Access youth account dashboard: Given the parent is logged into the banking web application When the parent navigates to the youth account section Then the system should display the youth account dashboard with current balance and recent activity
- Scenario 2 – Allocate funds to youth account: Given the parent is on the youth account dashboard When the parent selects "Add Funds" Then the system should allow the parent to transfer funds from their primary account to the youth account
- Scenario 3 – Set spending limit: Given the parent is viewing the youth account management section When the parent configures a weekly spending limit Then the system should save and display the limit on the dashboard
- Scenario 4 – View youth spending activity: Given the parent is viewing the youth account dashboard When the parent selects the activity section Then the system should display recent transactions made from the youth account
- Scenario 5 – Handle insufficient balance during fund allocation: Given the parent attempts to transfer funds When the selected parent account does not have sufficient balance Then the system should display an error message indicating insufficient funds

UI Tasks:
- SCIB-195: Design youth account dashboard layout
- SCIB-194: Design fund transfer interaction for youth account
- SCIB-196: Design spending limit configuration interface
- SCIB-197: Design youth account activity view

8. API SUMMARY

Available APIs:
- GET /api/youth-accounts/{youthAccountId}/dashboard → Fetch dashboard data including balance, spending limit, and recent transactions
- POST /api/youth-accounts/{youthAccountId}/fund-transfer → Transfer funds from parent account to youth account
- PUT /api/youth-accounts/{youthAccountId}/spending-limit → Set or update weekly spending limit
- GET /api/youth-accounts/{youthAccountId}/transactions → List youth account transactions with pagination and filtering

Authentication:
- OAuth2 with parent/guardian role required
- All endpoints require proper authorization

Data Models:
- YouthAccountDashboard: accountBalance, spendingLimit, currency, recentTransactions
- FundTransferRequest: sourceAccountId, transferAmount, currency
- FundTransferResponse: transferId, status, updatedBalance
- SpendingLimitRequest: weeklyLimit
- SpendingLimitResponse: youthAccountId, weeklyLimit, status
- Transaction: transactionId, date, merchant, amount, transactionType
- TransactionsResponse: transactions[], pagination

- DashboardPageComponent → GET /api/youth-accounts/{youthAccountId}/dashboard

13. INTERACTION FLOWS

User Action → UI → API:

1. View Dashboard:
   - User navigates to dashboard
   → DashboardPageComponent loads
   → Call GET /api/youth-accounts/{id}/dashboard
   → Display balance, limit, and recent activity

2. Add Funds:
   - User clicks "Add Funds" button
   → Navigate to FundTransferPageComponent
   → User selects source account and enters amount
   → User clicks "Transfer"
   → Show TransferConfirmationModalComponent
   → User confirms
   → Call POST /api/youth-accounts/{id}/fund-transfer
   → Show success message and updated balance

3. Set Spending Limit:
   - User clicks "Set Limit" or "Edit" button
   → Navigate to SpendingLimitPageComponent
   → User enters new limit amount
   → User clicks "Save"
   → Call PUT /api/youth-accounts/{id}/spending-limit
   → Show success confirmation
   → Return to dashboard with updated limit

4. View Activity:
   - User clicks "View Activity" or "View All"
   → Navigate to ActivityPageComponent
   → Call GET /api/youth-accounts/{id}/transactions
   → Display paginated transaction list
   → User can filter by date range
   → Update API call with filter parameters

5. Handle Errors:
   - API returns error response
   → Display appropriate error component
   → User clicks "Retry"
   → Repeat failed API call
   → User clicks "Cancel" or "Back"
   → Return to previous state
